# 🧩 防災微弱訊號「碎形通報」積木元件

這是一個專門為了極端災場（比如土石流、堰塞湖爆發，現場手機網路只剩一格）設計的快速通報網頁元件。

它最大的特色就是：**不跟災民閒聊，也不讓你填大表單。每點一題、就送一題、後台就同步一題。** 就算災民點到一半手機斷電，前面已經選好的救命資訊和定位也早就傳到指揮中心了。

## 🌊 2. 天災現場的三大痛點 (Why we build this?)

* **傳統App太肥：** 災區網路只剩一格，開個網頁要載入幾MB的地圖或圖片，畫面直接卡死。
* **表單一次送出很賭運氣：** 傳統系統非得要把姓名、電話、災情全部填完才能按「送出」。在訊號斷續的現場，傳輸到一半斷線，就是整包失敗，後台什麼都拿不到。
* **個資法卡死數據共享：** 收集姓名電話會碰觸到《個資法》，導致消防局、台電、水利署之間沒辦法在第一時間直接共享這份資料。

---

## 🚀 3. 我們的解決方案 (Our Solution)

* **⚡ 碎形化增量 Commit 協定：** 拋棄傳統表單。把資料拆成極小的碎片，點擊按鈕的瞬間就以 `PATCH` 方式異步射上雲端。
* **🎨 100% 系統原生 Emoji 渲染：** 全網頁「零圖片」，全部用手機內建的 Unicode 符號（如 `🔴`, `💀`, `🌊`）來排版。每個圖標只要 2 幾位元組，絕對不吃頻寬、而且在任何手機上都絕不破圖。
* **🔒 去識別化隱私設計：** 系統開頭就隨機生成一串 UUID 當作臨時身分，不抓個資、不問電話。沒有個資外洩問題，政府跨部會就能無縫、合法地秒級共享數據。

## 📱 5. 問題鏈設計與後台派工邏輯 (Use Case)

我們設計的每一個問題和選項，都有對應的後台救災派遣邏輯，不浪費災民任何一秒的頻寬：

### 🔴 路線 A：有危險急需救援（消防特搜派工鏈）

* **Q1：目前有立即生命危險嗎？** ──► 【決策價值】 決定搜救優先級。選 `💀 有` 的封包一進後台，系統立刻亮最高紅燈，讓特搜隊直升機或搜救艇列為第一順位出動。
* **Q2：現場有行動不便/重症者嗎？** ──► 【決策價值】 精準調配醫療資源。答 `🩸 外傷` 提示特搜隊帶 EMT-P 醫療包；答 `🏥 洗腎` 代表需要電力，後台會立刻連動台電組件優先搶修該區變壓器。
* **Q3：現場大約有幾人受困？** ──► 【決策價值】 決定載具運量。1人派皮卡車，`20人以上`立刻觸發指揮中心警報，改調派中型戰術輪車並預備收容所物資。
* **Q4：您目前受困在哪種環境？** ──► 【決策價值】 決定搜救工具。`🏠 屋頂` 派吊掛或衝鋒艇；`🚗 車內` 派攜帶油壓剪之消防車；`🌳 山坡` 派繩索專精特搜隊。

### 🟡 路線 B：看到災情但目前安全（跨部會搶修鏈）

* **Q1：看到了哪種主要災情？** ──► 【決策價值】 自動化派工。選 `🌊 大水` 丟給水利署；選 `🪨 山崩` 丟給公路局；選 `⚡ 停電` 自動派單給台電。
* **Q2：周邊交通通行狀況如何？** ──► 【決策價值】 答 `❌ 完全阻斷` 則後台 GIS 地圖自動封鎖該區道路，提醒後方救援車輛繞道，並派推土機前導開路。
* **Q3：遭受破壞的主要地方是？** ──► 評估受災主體（民宅、主要橋樑、農田魚塭），決定物資撥發方向。
* **Q4：災情似乎正在？** ──► 【決策價值】 災情趨勢動態預測。短時間若有多筆 `📈 持續擴大` 回報，系統會將該下游列為潛在孤島，啟動預防性強制撤離。

### 🟢 路線 C：目前平安 / 周圍無災

* **核心效益：** 點擊瞬間上傳坐標報平安，同時前端直接渲染免頻寬的「離線自救告示牌」，提醒民眾切勿前往低窪地區或溪畔看熱鬧。

## 🏆 6. 專案亮點與後續擴充 (Highlights)

* **頻寬耐受度提升 30,000 倍：** 單次碎形微封包只有約 500 Bytes（傳一張現場照片要 15MB 左右）。極端災場裡網路再爛，它也能保證 100% 傳送成功。
* **零成本熱插拔擴充：** 雲端採用無 Schema 結構。未來政府如果想要對不同天災新增提問，在不用修改資料庫結構、不改程式碼的前提下，一秒就能直接線上更新問題鏈。
* **未來的 AI 燈號群聚分析：** 本元件產出的標準化 JSON 流，未來能直接無縫熱插拔對接大數據 AI 模組，在 3 秒內自動圈選出「隱形重災區熱點」，提供給指揮官最直覺的視覺化調度參考。
